About

Built in Serbia.
Working across
Scandinavia.

We started meeha in Novi Sad because we kept seeing the same thing: fast-growing companies with data teams that were drowning. Not because the engineers weren't good — but because nobody had built the foundation properly in the first place.

Most of our clients are Swedish, Norwegian, and Danish companies in the Series A–C range — past the scrappy startup phase, not yet big enough to build a 10-person data platform team. That gap is exactly where we work.

We're remote by default, async-friendly, and we communicate in plain English (and occasionally broken Swedish). We work in your timezone, show up to your standups, and we don't disappear after the handover.

How we build

Our engineering
standards

Inspired by the Software Craftsmanship Manifesto. These aren't aspirations — they're non-negotiables on every project.

It isn't done until it's tested

Every pipeline ships with unit tests, integration tests, and data quality checks. If there's no test coverage, it doesn't go to production. Period.

dbt tests Great Expectations pytest Soda

Code quality is not optional

We follow PEP 8, use linters (ruff, flake8), enforce type hints, and run pre-commit hooks. Readable code is maintainable code — and your team has to live with it after we leave.

ruff mypy pre-commit black

Everything is version controlled

Code, SQL, infrastructure, configs — all in Git. No snowflake environments (the bad kind). No undocumented manual steps. If it's not in the repo, it doesn't exist.

Git Terraform CI/CD GitOps

Observability from day one

Pipelines have logging, alerting, and lineage from the first deploy. You know what ran, when it ran, how long it took, and what broke — without having to ask us.

OpenLineage Monte Carlo Airflow logs PagerDuty

Incremental over full refresh

We design for efficiency by default. Incremental models, partition pruning, smart backfills. Your cloud bill matters — we treat it like our own money.

Incremental models Partitioning Cost monitoring

Done means documented

Every project ships with README files, dbt docs, architecture diagrams, and runbooks. The next engineer — including your own team — should be able to onboard without calling us.

dbt docs Confluence Architecture diagrams
